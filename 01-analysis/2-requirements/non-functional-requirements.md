# Нефункциональные требования — «Кулинарная студия»

## Назначение документа

Документ фиксирует нефункциональные требования к клиентскому мобильному приложению кулинарной студии «Шеф-стол».

Нефункциональные требования описывают не отдельные функции, а качество работы приложения: удобство, понятность, адаптацию под мобильные устройства, обработку ошибок, безопасность пользовательских данных и поддерживаемость решения.

---

## 1. Границы нефункциональных требований

В рамках текущей поставки нефункциональные требования относятся к:

- клиентскому мобильному приложению;
- API-контракту между клиентским приложением и существующим бэкендом;
- отображению данных о кулинарных классах;
- форме записи;
- обработке ошибок и пустых состояний;
- работе с пользовательскими данными на стороне клиентского приложения.

В рамках текущей поставки не описываются:

- внутренняя архитектура существующего бэкенда;
- физическая база данных;
- серверная транзакционность;
- серверное SLA;
- админка;
- интерфейс владельца;
- интерфейс шефа;
- интеграции с платёжными системами.

## 2. Категории требований

| Категория | Описание |
|---|---|
| Удобство использования | Насколько легко клиенту понять приложение и записаться на класс. |
| Мобильная адаптация | Насколько корректно приложение работает на экране телефона. |
| Производительность | Насколько быстро приложение реагирует на действия пользователя. |
| Надёжность и ошибки | Насколько понятно приложение ведёт себя при сбоях и пустых данных. |
| Безопасность данных | Как приложение обращается с персональными данными клиента. |
| Доступность интерфейса | Насколько интерфейс читаемый и понятный для разных пользователей. |
| Поддерживаемость | Насколько удобно развивать и изменять проект дальше. |

## 3. Удобство использования

| ID | Требование | Критерий проверки | Приоритет | Комментарий |
|---|---|---|---|---|
| NFR-001 | Основной сценарий записи должен быть понятен клиенту без дополнительной инструкции. | Пользователь может пройти путь: список классов → карточка класса → форма записи → подтверждение. | Высокий | Это главный пользовательский путь MVP. |
| NFR-002 | Названия кнопок и сообщений должны быть понятными и связанными с кулинарной студией. | В интерфейсе используются формулировки вроде «Записаться», «Мест нет», «Класс отменён студией», а не технические сообщения. | Высокий | Клиент не должен видеть внутренние технические термины. |
| NFR-003 | Ошибки в форме записи должны показываться рядом с соответствующими полями. | Если клиент не заполнил обязательное поле, приложение показывает, какое именно поле нужно исправить. | Высокий | Это снижает количество неуспешных отправок формы. |
| NFR-004 | После успешной записи клиент должен получить понятное подтверждение. | После отправки формы отображается сообщение об успешной записи и статус записи. | Высокий | Клиент должен понимать, что действие завершено. |
| NFR-005 | Пустые состояния должны объяснять ситуацию пользователю. | Если классов нет, клиент видит текст вроде «Пока нет доступных классов», а не пустой экран. | Высокий | Важно для доверия к приложению. |

## 4. Мобильная адаптация

| ID | Требование | Критерий проверки | Приоритет | Комментарий |
|---|---|---|---|---|
| NFR-006 | Интерфейс должен быть рассчитан прежде всего на использование с телефона. | Основные экраны корректно отображаются в мобильной ширине, без горизонтальной прокрутки. | Высокий | В брифе указано, что пользоваться будут в основном с телефона. |
| NFR-007 | Карточки классов должны оставаться читаемыми на небольшом экране. | Название, дата, время, цена и свободные места не сливаются и не перекрывают друг друга. | Высокий | Список классов — первый экран клиента. |
| NFR-008 | Форма записи должна быть удобной для заполнения на телефоне. | Поля формы расположены последовательно, подписи понятны, кнопка отправки видна после заполнения данных. | Высокий | Запись не должна требовать сложных действий. |
| NFR-009 | Интерактивные элементы должны быть достаточно крупными для нажатия на сенсорном экране. | Кнопки и поля формы можно нажать без высокой точности попадания. | Средний | Это улучшает удобство мобильного интерфейса. |

## 5. Производительность

| ID | Требование | Критерий проверки | Приоритет | Комментарий |
|---|---|---|---|---|
| NFR-010 | Приложение должно показывать состояние загрузки при получении данных. | Пока список классов или запись загружаются, пользователь видит индикатор или текст загрузки. | Высокий | Пользователь должен понимать, что действие выполняется. |
| NFR-011 | После получения данных от API интерфейс должен обновляться без ручной перезагрузки страницы. | Список классов, карточка и статус записи обновляются в интерфейсе после ответа системы. | Высокий | Это базовое ожидание от клиентского приложения. |
| NFR-012 | Приложение не должно блокировать весь интерфейс при загрузке одного блока данных. | Если загружается карточка класса или статус записи, пользователь видит локальное состояние загрузки. | Средний | Требование важно для ощущения стабильности приложения. |
| NFR-013 | Клиентское приложение должно избегать лишней перезагрузки одинаковых данных в рамках одного пользовательского сценария. | При переходе между списком и карточкой уже полученные данные используются повторно, если они актуальны. | Средний | Это снижает задержки и нагрузку на API. |

## 6. Надёжность и обработка ошибок

| ID | Требование | Критерий проверки | Приоритет | Комментарий |
|---|---|---|---|---|
| NFR-014 | При отсутствии интернет-соединения приложение должно показывать понятное сообщение. | Пользователь видит, что действие не выполнено из-за проблемы соединения. | Высокий | Нельзя оставлять пользователя без объяснения. |
| NFR-015 | При недоступности API приложение должно показывать понятную ошибку. | Вместо технического ответа клиент видит сообщение, что данные временно не удалось загрузить. | Высокий | Пользователь не должен видеть сырые технические ошибки. |
| NFR-016 | Если запись не создана, приложение не должно показывать успешное подтверждение. | При отказе API клиент видит ошибку, а не статус успешной записи. | Высокий | Особенно важно при отсутствии свободных мест. |
| NFR-017 | Приложение должно корректно обрабатывать ситуацию, когда свободные места закончились во время заполнения формы. | Клиент видит сообщение, что мест уже нет, и запись не считается созданной. | Высокий | Бэкенд остаётся источником истины по местам. |
| NFR-018 | Если часть необязательных данных не пришла через API, интерфейс не должен полностью ломаться. | Приложение показывает доступные данные и не падает из-за отсутствия необязательного поля. | Средний | Например, если отсутствует описание шефа или подробное меню. |

## 7. Безопасность и персональные данные

| ID | Требование | Критерий проверки | Приоритет | Комментарий |
|---|---|---|---|---|
| NFR-019 | Приложение должно собирать только те персональные данные, которые нужны для записи. | В форме нет лишних полей, не связанных с записью на класс. | Высокий | Минимизация данных снижает риски. |
| NFR-020 | Персональные данные клиента не должны отображаться в интерфейсе без необходимости. | После отправки формы приложение показывает только нужную информацию о записи. | Высокий | Особенно важно для телефона, которым могут пользоваться в публичном месте. |
| NFR-021 | Информация об аллергиях должна обрабатываться как чувствительная пользовательская информация. | Поле аллергий не используется в интерфейсе без необходимости и передаётся только в составе записи. | Высокий | Аллергии относятся к личной информации клиента. |
| NFR-022 | Приложение не должно сохранять реальные персональные данные в демонстрационном коде. | В учебной реализации используются тестовые данные, а не реальные данные клиентов. | Высокий | Это важно для безопасной демонстрации проекта. |
| NFR-023 | Сообщения об ошибках не должны раскрывать внутренние технические детали. | Клиент видит понятное сообщение, а не стек ошибки, внутренний код сервера или служебные данные. | Высокий | Это снижает риск утечки технической информации. |

## 8. Доступность и читаемость интерфейса

| ID | Требование | Критерий проверки | Приоритет | Комментарий |
|---|---|---|---|---|
| NFR-024 | Текст в интерфейсе должен быть читаемым на мобильном экране. | Основные надписи, цены, даты и статусы можно прочитать без увеличения страницы. | Высокий | Клиент должен быстро понять информацию о классе. |
| NFR-025 | Статусы записи должны быть визуально и текстово различимы. | Статусы «Подтверждена», «Отменена клиентом», «Отменён студией» не выглядят одинаково. | Средний | Это снижает риск неправильного понимания записи. |
| NFR-026 | Обязательные поля формы должны быть явно обозначены. | Клиент до отправки понимает, какие поля нужно заполнить обязательно. | Высокий | Это уменьшает количество ошибок при записи. |
| NFR-027 | Ошибки должны быть написаны простым языком. | Сообщения объясняют, что исправить: например, «Введите имя», «Укажите телефон», «Мест уже нет». | Высокий | Пользователь не должен расшифровывать технический текст. |

| ID | Требование | Критерий проверки | Приоритет | Комментарий |
|---|---|---|---|---|

| NFR-030 | Данные в учебной реализации могут быть тестовыми, но должны соответствовать предметной области. | В коде используются классы, шефы, программы, даты и статусы, похожие на реальные данные кулинарной студии. | Высокий | Демонстрация должна выглядеть связанной с брифом. |

## 12. Итог

Нефункциональные требования MVP направлены на то, чтобы клиентское приложение было:

1. понятным для клиента;
2. удобным на мобильном экране;
3. устойчивым к ошибкам загрузки и отправки данных;
4. аккуратным в работе с персональными данными;
5. простым для демонстрации и дальнейшей доработки.

Основной акцент сделан на качестве клиентского сценария: клиент должен без лишних объяснений найти подходящий кулинарный класс, записаться на него и получить понятный статус записи или понятную ошибку.
