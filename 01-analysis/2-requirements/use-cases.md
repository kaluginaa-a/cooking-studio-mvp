# Use Cases — «Кулинарная студия»

## Назначение документа

Документ описывает основные пользовательские сценарии клиентского приложения кулинарной студии «Шеф-стол».

Use Case показывает не только то, что хочет пользователь, но и как именно он взаимодействует с приложением шаг за шагом. В отличие от User Story, сценарий включает основной успешный путь, альтернативные потоки и ошибки.

---

## 1. Границы сценариев

В рамках текущей поставки рассматривается только роль клиента.

Клиентское приложение:

- показывает доступные кулинарные классы;
- показывает подробную карточку класса;
- позволяет оформить запись на класс;
- передаёт данные клиента через API;
- отображает статус записи;
- показывает ошибки, если запись невозможна;
- отображает отмену класса студией.

Клиентское приложение не выполняет следующие действия:

- не создаёт расписание;
- не редактирует программы классов;
- не назначает шефов;
- не управляет закупками продуктов;
- не отменяет класс от имени студии;
- не реализует интерфейс владельца, администратора или шефа.

---

## 2. Участники

| Участник | Описание |
|---|---|
| Клиент | Пользователь, который выбирает кулинарный класс и записывается на него. |
| Клиентское приложение | Интерфейс, через который клиент просматривает классы и оформляет запись. |
| API существующего бэкенда | Источник данных о классах, свободных местах, шефах и статусах записей. |

# UC-001. Просмотр списка доступных кулинарных классов

## Основная информация

| Поле | Значение |
|---|---|
| ID | UC-001 |
| Название | Просмотр списка доступных кулинарных классов |
| Актор | Клиент |
| Цель | Выбрать подходящий класс по дате, времени, программе и свободным местам |
| Триггер | Клиент открывает приложение или раздел со списком классов |
| Связанные User Stories | US-001, US-002, US-011 |
| Связанные бизнес-требования | BR-001, BR-003, BR-004 |

## Предусловия

- Клиент открыл приложение.
- У клиента есть доступ к интернету.
- Существующий бэкенд может отдавать список доступных классов через API.

## Основной успешный сценарий

1. Клиент открывает приложение.
2. Приложение запрашивает список доступных кулинарных классов.
3. Бэкенд возвращает классы на ближайшие 7 дней.
4. Приложение отображает список классов.
5. В карточке каждого класса клиент видит основные данные: программу, дату, время, шефа, цену и свободные места.
6. Клиент выбирает интересующий класс для просмотра подробностей.

## Альтернативные потоки

### A1. На ближайшие 7 дней нет доступных классов

1. Клиент открывает список классов.
2. Бэкенд возвращает пустой список.
3. Приложение показывает сообщение: «Пока нет доступных классов».
4. Клиент остаётся на экране списка.

### A2. Часть данных о классе недоступна

1. Бэкенд возвращает список классов, но по некоторым полям нет данных.
2. Приложение показывает доступные данные.
3. Недостающие необязательные данные не должны ломать отображение списка.

## Ошибки и исключения

| ID | Ситуация | Поведение приложения |
|---|---|---|
| E-001.1 | Нет соединения с интернетом | Показать сообщение об ошибке загрузки и предложить повторить попытку |
| E-001.2 | API временно недоступен | Показать понятное сообщение, что список классов сейчас не удалось загрузить |
| E-001.3 | Получен некорректный ответ API | Не показывать пустой экран без объяснения, вывести сообщение об ошибке |

## Постусловия

- Клиент видит список доступных классов.
- Если классов нет или произошла ошибка, клиент видит понятное состояние интерфейса.

# UC-002. Просмотр подробной карточки класса

## Основная информация

| Поле | Значение |
|---|---|
| ID | UC-002 |
| Название | Просмотр подробной карточки кулинарного класса |
| Актор | Клиент |
| Цель | Получить подробную информацию о занятии перед записью |
| Триггер | Клиент нажимает на класс в списке |
| Связанные User Stories | US-003, US-011 |
| Связанные бизнес-требования | BR-003, BR-004 |

## Предусловия

- Клиент находится на экране списка классов.
- В списке есть хотя бы один доступный класс.
- Данные о выбранном классе доступны через API или уже загружены в приложении.

## Основной успешный сценарий

1. Клиент выбирает класс из списка.
2. Приложение открывает подробную карточку класса.
3. Приложение показывает программу класса, меню, уровень сложности, дату, время, длительность, шефа, цену, адрес студии и количество свободных мест.
4. Клиент изучает информацию.
5. Если класс доступен, клиент может нажать кнопку «Записаться».

## Альтернативные потоки

### A1. Класс уже заполнен

1. Клиент открывает карточку класса.
2. Приложение получает информацию, что свободных мест нет.
3. Кнопка записи становится недоступной или заменяется сообщением «Мест нет».

### A2. Класс отменён студией

1. Клиент открывает карточку класса.
2. Приложение получает статус «Отменён студией».
3. Приложение показывает причину отмены, если она передана через API.
4. Повторная запись на этот класс недоступна.

## Ошибки и исключения

| ID | Ситуация | Поведение приложения |
|---|---|---|
| E-002.1 | Данные выбранного класса не найдены | Показать сообщение, что класс недоступен или был удалён из расписания |
| E-002.2 | Ошибка загрузки подробностей | Показать сообщение об ошибке и кнопку повторной загрузки |
| E-002.3 | Свободные места изменились | Показать актуальное состояние, полученное от API |

## Постусловия

- Клиент получил подробную информацию о классе.
- Если запись возможна, клиент может перейти к форме записи.
- Если запись невозможна, приложение показывает понятную причину.

# UC-003. Запись клиента на кулинарный класс

## Основная информация

| Поле | Значение |
|---|---|
| ID | UC-003 |
| Название | Запись клиента на кулинарный класс |
| Актор | Клиент |
| Цель | Оформить участие в выбранном классе |
| Триггер | Клиент нажимает кнопку «Записаться» в карточке класса |
| Связанные User Stories | US-004, US-005, US-006, US-007, US-008, US-011 |
| Связанные бизнес-требования | BR-001, BR-002, BR-003, BR-006, BR-007 |

## Предусловия

- Клиент открыл карточку класса.
- Класс доступен для записи.
- На классе есть свободные места.
- Приложение может отправить запрос на создание записи через API.

## Основной успешный сценарий

1. Клиент нажимает кнопку «Записаться».
2. Приложение открывает форму записи.
3. Клиент вводит имя и контактные данные.
4. Клиент указывает информацию об аллергиях, если это предусмотрено формой.
5. Клиент выбирает, придёт ли со своим фартуком и набором ножей или воспользуется комплектом студии.
6. Клиент при необходимости оставляет комментарий.
7. Клиент нажимает кнопку отправки формы.
8. Приложение проверяет обязательные поля.
9. Приложение отправляет запрос на создание записи.
10. Бэкенд проверяет наличие свободных мест.
11. Бэкенд создаёт запись и возвращает успешный ответ.
12. Приложение показывает подтверждение и статус записи.

## Альтернативные потоки

### A1. Клиент не указал аллергию

1. Клиент оставляет поле аллергий пустым.
2. Если поле необязательное, форма отправляется.
3. Если поле обязательное, приложение просит указать информацию или выбрать вариант «аллергий нет».

### A2. Клиент выбирает комплект студии

1. Клиент выбирает, что ему нужен фартук и набор ножей от студии.
2. Приложение передаёт этот выбор в составе записи.
3. Студия получает информацию для подготовки занятия.

### A3. Клиент приходит со своим инвентарём

1. Клиент выбирает, что придёт со своим фартуком и набором ножей.
2. Приложение передаёт этот выбор в составе записи.
3. Комплект студии для клиента не требуется.

## Ошибки и исключения

| ID | Ситуация | Поведение приложения |
|---|---|---|
| E-003.1 | Не заполнено обязательное поле | Показать ошибку рядом с полем и не отправлять форму |
| E-003.2 | Свободные места закончились во время заполнения формы | Показать сообщение, что мест уже нет, и не создавать ложное подтверждение |
| E-003.3 | API вернул отказ при создании записи | Показать причину отказа, если она передана |
| E-003.4 | Нет сети во время отправки формы | Показать сообщение, что запись не удалось отправить |
| E-003.5 | Данные формы заполнены в неверном формате | Подсветить поля, которые нужно исправить |

## Постусловия

- При успешном сценарии запись создана.
- Клиент видит статус записи.
- При ошибке запись не создаётся, а клиент получает понятное сообщение.

# UC-004. Просмотр статуса записи

## Основная информация

| Поле | Значение |
|---|---|
| ID | UC-004 |
| Название | Просмотр статуса записи |
| Актор | Клиент |
| Цель | Понять, подтверждено ли участие в классе |
| Триггер | Клиент открывает экран подтверждения или свою запись |
| Связанные User Stories | US-008, US-010, US-011 |
| Связанные бизнес-требования | BR-007, BR-008 |

## Предусловия

- У клиента есть созданная запись.
- Приложение может получить статус записи через API.
- Способ идентификации клиента или записи должен быть определён в API-контракте.

## Основной успешный сценарий

1. Клиент открывает экран записи.
2. Приложение запрашивает актуальные данные записи.
3. Бэкенд возвращает статус записи.
4. Приложение показывает клиенту статус: подтверждена, отменена клиентом, отменена студией или завершена.
5. Если есть дополнительная информация, приложение показывает её клиенту.

## Альтернативные потоки

### A1. Запись отменена студией

1. Клиент открывает запись.
2. Приложение получает статус «Отменён студией».
3. Приложение показывает статус и причину отмены, если она передана.

### A2. Запись уже завершена

1. Клиент открывает запись после окончания класса.
2. Приложение показывает статус «Завершена».
3. Если функция оценки шефа включена в релиз, клиент может перейти к оценке.

## Ошибки и исключения

| ID | Ситуация | Поведение приложения |
|---|---|---|
| E-004.1 | Запись не найдена | Показать сообщение, что запись не найдена |
| E-004.2 | Нет доступа к записи | Показать сообщение, что запись недоступна |
| E-004.3 | Ошибка загрузки статуса | Показать сообщение об ошибке и предложить повторить попытку |

## Постусловия

- Клиент видит актуальный статус записи.
- Если статус получить не удалось, клиент видит понятное сообщение.

# UC-005. Отмена записи клиентом

## Основная информация

| Поле | Значение |
|---|---|
| ID | UC-005 |
| Название | Отмена записи клиентом |
| Актор | Клиент |
| Цель | Отменить участие в классе заранее |
| Триггер | Клиент нажимает кнопку «Отменить запись» |
| Связанные User Stories | US-009, US-008, US-011 |
| Связанные бизнес-требования | BR-005, BR-007 |

## Предусловия

- У клиента есть активная запись.
- Запись ещё не отменена.
- Класс ещё не завершён.
- Правило допустимой отмены должно быть уточнено заказчиком.

## Основной успешный сценарий

1. Клиент открывает свою запись.
2. Клиент нажимает кнопку «Отменить запись».
3. Приложение показывает подтверждение действия.
4. Клиент подтверждает отмену.
5. Приложение отправляет запрос на отмену записи.
6. Бэкенд проверяет, допустима ли отмена по времени.
7. Бэкенд меняет статус записи на «Отменена клиентом».
8. Приложение показывает клиенту обновлённый статус.

## Альтернативные потоки

### A1. Клиент передумал отменять запись

1. Клиент нажимает «Отменить запись».
2. Приложение показывает подтверждение.
3. Клиент выбирает «Не отменять».
4. Статус записи не меняется.

### A2. Отмена слишком поздняя

1. Клиент пытается отменить запись незадолго до начала класса.
2. Бэкенд возвращает отказ или специальный статус.
3. Приложение показывает сообщение, что отмена уже ограничена правилами студии.

## Ошибки и исключения

| ID | Ситуация | Поведение приложения |
|---|---|---|
| E-005.1 | Запись уже отменена | Показать актуальный статус записи |
| E-005.2 | Запись не найдена | Показать сообщение, что запись не найдена |
| E-005.3 | Ошибка сети при отмене | Показать сообщение, что отмену не удалось выполнить |
| E-005.4 | Бэкенд отказал в отмене | Показать причину отказа, если она передана |

## Постусловия

- При успешной отмене статус записи изменён на «Отменена клиентом».
- Если отмена невозможна, статус записи не меняется, а клиент видит причину.

# UC-006. Получение информации об отмене класса студией

## Основная информация

| Поле | Значение |
|---|---|
| ID | UC-006 |
| Название | Отображение отмены класса студией |
| Актор | Клиент |
| Цель | Узнать, что класс не состоится |
| Триггер | Клиент открывает запись или получает обновление статуса |
| Связанные User Stories | US-010, US-008, US-011 |
| Связанные бизнес-требования | BR-007, BR-008 |

## Предусловия

- У клиента есть запись на класс.
- Класс отменён студией в существующей инфраструктуре.
- Бэкенд передаёт в клиентское приложение статус отмены.

## Основной успешный сценарий

1. Студия отменяет класс во внешней системе.
2. Бэкенд обновляет статус связанных записей.
3. Клиент открывает приложение или экран своей записи.
4. Приложение получает актуальный статус записи.
5. Приложение показывает статус «Отменён студией».
6. Если причина отмены передана, приложение показывает её клиенту.
7. Приложение не позволяет повторно записаться на отменённый класс.

## Альтернативные потоки

### A1. Причина отмены не передана

1. Клиент открывает запись.
2. Приложение получает статус «Отменён студией» без причины.
3. Приложение показывает общий текст о том, что класс отменён студией.

### A2. Клиент пытается записаться на отменённый класс

1. Клиент открывает карточку отменённого класса.
2. Приложение показывает статус отмены.
3. Кнопка записи недоступна.

## Ошибки и исключения

| ID | Ситуация | Поведение приложения |
|---|---|---|
| E-006.1 | Статус записи не удалось обновить | Показать сообщение об ошибке и оставить последний известный статус |
| E-006.2 | Причина отмены отсутствует | Показать общий текст без причины |
| E-006.3 | Клиент пытается создать запись на отменённый класс | Показать сообщение, что запись на этот класс невозможна |

## Постусловия

- Клиент видит, что класс отменён студией.
- Запись не удаляется из интерфейса.
- Повторная запись на отменённый класс недоступна.

## 4. Открытые вопросы

Перед фиксацией финальных требований нужно уточнить:

1. Какие поля формы записи обязательны.
2. Может ли клиент записать несколько участников одной заявкой.
3. Является ли выбор по фартуку и набору ножей обязательной частью MVP.
4. Какой срок отмены записи считается допустимым.
5. Как клиент идентифицируется для просмотра своих записей.
6. Какие статусы записи передаются через API.
7. Какие ошибки API должны обрабатываться на стороне клиента.
8. Входит ли оценка шефа в первую версию.

## Итог

Ключевой сценарий MVP — самостоятельная запись клиента на кулинарный класс.

Минимальная цепочка взаимодействия выглядит так:

1. клиент открывает список доступных классов;
2. выбирает подходящий класс;
3. открывает подробную карточку;
4. заполняет форму записи;
5. получает подтверждение или понятную ошибку;
6. видит актуальный статус записи.

Дополнительные сценарии — отмена записи клиентом, отображение отмены класса студией, указание аллергий и выбор по фартуку/набору ножей — уточняются и включаются в MVP при подтверждении заказчиком.
